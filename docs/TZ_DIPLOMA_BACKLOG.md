# ТЗ-черновик: дипломный функционал (A + B) — сделано / осталось

Документ для доработки в отдельном чате (ИИ / руководитель).  
**Тема (примерно):** разработка кроссплатформенного мобильного приложения для мониторинга психоэмоционального состояния **с адаптивным прогнозированием и персонализацией рекомендаций**.

**Цель по продукту:** и **качественный UI с понятной аналитикой**, и **модель прогноза с метриками** (для главы про эксперимент / оценку качества).

---

## 1. Уровни

| Уровень | Содержание |
|--------|------------|
| **A (минимум защиты)** | Дневник + аналитика + **одна модель прогноза на сервере** + **персонализированные советы** по прогнозу и профилю пользователя |
| **B (сильный диплом)** | A + **объяснение прогноза** + **обратная связь по советам** + **пересчёт (ретренинг) модели** по расписанию и/или по числу новых записей |

---

## 2. Уже реализовано (текущий репозиторий)

### 2.1. Клиент (Flutter)

- Навигация: дашборд, добавление записи, история, советы; настройки (язык ru/en, светлая/тёмная тема).
- **Дневник / запись:** настроение, стресс, энергия; заметка; дата; категория; сон (часы); активность (минуты).
- **Дашборд:** индекс благополучия (с сервера), график настроения, цели/стрик, сравнение недель, последняя запись, быстрое дублирование записи.
- **История:** список, фильтры, аналитика (графики при достаточном числе точек), редактирование/удаление.
- **Советы:** блок «для вас» с API; локальные советы по категориям; избранное; таймер дыхания 4–7–8; быстрая запись прогулки.
- **UI/UX:** единая сетка отступов, M3, шрифты (Inter + Plus Jakarta Sans), шиммеры загрузки, тёмная тема, конфиг base URL (`AppConfig` + `dart-define`), `intl` для дат.
- **Связь с API:** `ApiService` (CRUD настроения, wellbeing, рекомендации, health).

### 2.2. Сервер (Python / FastAPI)

- **CRUD** записей настроения: `POST/GET/PUT/DELETE` `/mood`, `GET /mood/{id}`.
- **Индекс благополучия:** `GET /wellbeing` (формула по последней записи).
- **Рекомендации (эвристики):** `GET /recommendations?lang=` — текст + уровень + wellbeing_index, без ML-прогноза.
- **PostgreSQL** + SQLAlchemy, миграции/скрипты под новые поля при необходимости.
- **Служебно:** `GET /health`.

### 2.3. Частично пересекается с будущей «моделью», но не является прогнозом

- Правила в `/recommendations` задают **совет по последнему состоянию**, не **прогноз на будущее** и не **обучаемая модель**.
- Локальные советы в приложении **не завязаны** на серверный прогноз.

## 3. Нужно сделать для уровня A

### 3.1. Сервер: одна модель прогноза + персонализация советов

- **Формализовать задачу прогноза** (зафиксировать в ТЗ): что именно предсказываем (пример: «риск высокого стресса завтра», «ожидаемое среднее настроение на следующий день», бинарная/ordinal классификация и т.д.).
- **Признаки:** минимум из уже собираемых полей (лаги настроения/стресса/энергии, сон, активность, день недели и др.).
- **Обучение и инференс на сервере:** одна выбранная модель (например, sklearn: логрег / дерево / бустинг — достаточно для диплома).
- **Хранение артефактов модели** по пользователю или одна модель + персональные коэффициенты — решить в ТЗ (минимум: отдельная таблица «версия модели / параметры» или файлы на диске сервера).
- **Новый endpoint** (условно): `GET /forecast` или поле в расширенном `GET /recommendations`): прогноз + уверенность + привязка к `user_id` если появится multi-user (сейчас условно один пользователь — явно описать в ТЗ).
- **Персонализированный список советов:** правило ранжирования/фильтрации локального набора советов и/или генерация текстов по шаблонам на основе **прогноза + профиля** (например, частые паттерны по истории).

### 3.2. Клиент: отображение прогноза и «красивая» аналитика

- **Экран или секция «Прогноз»** (дашборд или отдельная вкладка): визуализация прогноза, понятные подписи, состояние загрузки/ошибки.
- **Связка советов с сервером:** список советов после запроса, завязанного на прогноз (замена или дополнение текущего смешанного экрана советов).
- **Графики:** усилить читаемость под диплом (легенды, подписи осей, период «7/30 дней») — уточнить в ТЗ, какие графики обязательны для главы «интерфейс».

### 3.3. Документация и воспроизводимость

- Описание модели, состава признаков, версии зависимостей ML в `requirements.txt`.
- Краткая инструкция запуска обучения (скрипт или Makefile).

## 4. Нужно сделать для уровня B (поверх A)

### 4.1. Объяснение прогноза (explainability)

- В ответе API: структурированное поле **«факторы»** (топ признаков по вкладу или текстовое объяснение шаблоном).
- В UI: блок **«Почему такой прогноз»** без перегруза текстом.

### 4.2. Обратная связь по советам

- Модель данных: сохранение оценки совета (полезно / бесполезно или 1–5), идентификатор совета, время.
- API: `POST` (или `PUT`) для отправки обратной связи.
- UI: кнопки на карточке совета, опционально локальный кэш при офлайн.

### 4.3. Адаптивное переобучение (ретренинг)

- **Триггеры:** по расписанию (cron / APScheduler) и/или после **N** новых записей пользователя.
- **Логика:** пересчёт обучающей выборки из БД, переобучение, сохранение новой версии, откат при деградации метрик (опционально в ТЗ).
- **Учёт обратной связи по советам** (если есть): как веса или отдельный признак — зафиксировать в ТЗ.

### 4.4. Метрики качества прогноза (для раздела «эксперимент»)

- Протокол оценки: train/test split по времени, метрики (MAE/RMSE для регрессии, accuracy/F1 для классификации).
- Возможность **экспорта** или **экрана «качество модели»** для демонстрации (хотя бы таблица последнего прогона обучения), либо сохранение метрик в БД при каждом ретренинге.

## 5. Кросс-требования (и UI, и модель)

| Направление | Что зафиксировать в итоговом ТЗ |
|-------------|----------------------------------|
| **Графики и UX** | Список обязательных экранов/виджетов, единый стиль, доступность подписей |
| **Модель** | Тип задачи, алгоритм, входы/выходы, частота обновления |
| **Эксперимент** | Датасет (реальные записи / синтетика), метрики, сравнение «до/после» или с базовым правилом |
| **Безопасность и этика** | Один пользователь vs несколько; хранение данных; отсутствие сторонних чувствительных источников без согласия |

## 6. Открытые вопросы для финального ТЗ (заполнить перед разработкой)

1. **Целевая задача прогноза:** регрессия или классификация? горизонт **24 ч / 7 дней**?
2. **Пользователи:** строго один аккаунт (локальный диплом) или заготовка под multi-user / логин?
3. **Минимальное число записей** в истории до первого прогноза (например, не показывать модель, пока &lt; 14 записей)?
4. **Где показывать прогноз:** только дашборд или отдельная вкладка?
5. **Нужен ли экспорт** (PDF/CSV) для приложения к пояснительной записке?
6. **Ретренинг:** только по расписанию раз в неделю или только после N записей, или оба варианта?

Попросите вторую модель:

1. Зафиксировать **единую терминологию** (прогноз / рекомендация / персонализация / адаптация модели).
2. Преобразовать разделы 3–4 в **пронумерованные требования** с критериями приёмки.
3. Согласовать **противоречия** курсового текста («локальное хранение» vs сервер) с выбранной архитектурой диплома.
4. Добавить **диаграмму архитектуры** (клиент — API — БД — модуль ML — планировщик ретренинга).
